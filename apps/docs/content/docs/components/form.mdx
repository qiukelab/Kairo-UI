## Usage

```tsx
import { useState } from 'react';
import { Form, Field, FieldLabel, FieldControl, FieldError, Input } from '@kairo-ui/react';

export function Example() {
  const [errors, setErrors] = useState<Record<string, string>>({});

  return (
    <Form
      errors={errors}
      onFormSubmit={async (values) => {
        const result = await createAccount(values);
        if (result.error) setErrors({ email: result.error });
      }}
    >
      <Field name="email">
        <FieldLabel>Email</FieldLabel>
        <FieldControl render={<Input type="email" />} required />
        <FieldError />
      </Field>
    </Form>
  );
}
```

`Form` renders a native `<form>` with `noValidate` set, so the browser's own
validation-bubble UI never appears тАФ an invalid `Field` shows its message
through its own `FieldError` part instead, styled like the rest of the page.
It carries no look of its own beyond that: no spacing or layout is imposed, so
arrange the `Field`s and buttons inside it however you like.

Pair it with `Field` (and that component's `FieldLabel`/`FieldControl`/
`FieldError` parts) to get id and ARIA wiring, per-field validation and error
display for free тАФ see `Field`'s own docs page for its anatomy.

### When validation runs

Each `Field` validates itself тАФ via its own `validate` function and any
native constraint attributes (`required`, `pattern`, ...) on its control.
`validationMode` decides *when* that check runs for every `Field` that
doesn't set its own:

- **`'onSubmit'`** (default) тАФ a field is checked when the form is submitted;
  once a submit has been attempted, it re-checks on every change afterwards,
  so its error clears the moment it's fixed.
- **`'onBlur'`** тАФ a field is checked as soon as it loses focus.
- **`'onChange'`** тАФ a field is checked on every keystroke or change.

Submitting with an invalid field cancels the submit, moves focus to the first
invalid field, and shows its error тАФ neither `onSubmit` nor `onFormSubmit`
below runs in that case.

### Server-side errors

This is Form's headline feature: once every field is valid on the client,
hand back an `errors` object keyed by each control's `name` тАФ typically set
from whatever your server or form action responded with тАФ and the matching
`Field`'s `FieldError` displays it automatically. No manual lookup is
required, and a `Field` clears its own entry the moment its value changes, so
a stale server error never lingers once the user starts fixing it.

### Reacting to a valid submit

Two props fire once every field has validated successfully, and they can be
combined:

- **`onSubmit`** тАФ the plain native submit event. Use it for a traditional
  submission (e.g. posting to a `<form action="...">`) that `Form` only gates
  on validity.
- **`onFormSubmit`** тАФ called with the fields' values already collected into a
  plain object keyed by `name`. It additionally calls `preventDefault()` on
  the submit event, so nothing posts natively тАФ use it to send the values
  yourself (`fetch`, a server function, ...).

`actionsRef` exposes a `validate(name?)` action for triggering validation
imperatively тАФ every field when called with no argument, or a single named
one тАФ without waiting for a submit.

## Rejecting an email server-side

Submitting `taken@example.com` simulates a server rejecting that address тАФ
`onFormSubmit` hands the rejection back through `errors`, and the `email`
field's `FieldError` shows it without any extra wiring.

## Props

| Prop             | Type                                             | Default      | Description                                                                                    |
| ---------------- | ------------------------------------------------ | ------------ | ------------------------------------------------------------------------------------------------ |
| `validationMode` | `'onSubmit' \| 'onBlur' \| 'onChange'`           | `'onSubmit'` | When fields validate тАФ see **When validation runs** above.                                       |
| `errors`         | `Record<string, string \| string[]>`             | тАФ            | Errors keyed by control `name`, typically from a server response тАФ see **Server-side errors**.   |
| `onSubmit`       | `(event: FormEvent<HTMLFormElement>) => void`    | тАФ            | Called on a valid native submit. See **Reacting to a valid submit**.                              |
| `onFormSubmit`   | `(values: Record<string, unknown>, eventDetails) => void` | тАФ  | Called on a valid submit with the fields' values collected by `name`; prevents the native submit. |
| `actionsRef`     | `RefObject<{ validate: (name?: string) => void } \| null>` | тАФ | Ref exposing an imperative `validate` action.                                                     |
| `className`      | `string`                                          | тАФ            | Merged with the base `kairo-form` class.                                                          |
| ...              | Additional props                                  | тАФ            | All other props (e.g. `id`, `name`) are forwarded to the underlying `<form>` element.              |
