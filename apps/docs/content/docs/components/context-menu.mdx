## Usage

```tsx
import {
  ContextMenu,
  ContextMenuTrigger,
  ContextMenuContent,
  ContextMenuItem,
} from '@kairo-ui/react';

export function Example() {
  return (
    <ContextMenu>
      <ContextMenuTrigger>Right-click this area</ContextMenuTrigger>
      <ContextMenuContent>
        <ContextMenuItem>Copy</ContextMenuItem>
        <ContextMenuItem>Paste</ContextMenuItem>
      </ContextMenuContent>
    </ContextMenu>
  );
}
```

`ContextMenu` groups the parts (it renders no DOM element of its own — the
same as Base UI's `ContextMenu.Root`). `ContextMenuTrigger` is the area that
becomes right-clickable (or long-press-able on touch) — wrap whatever content
should own the menu in it; it stays visually unstyled beyond a `context-menu`
cursor hint, since it wraps arbitrary consumer content (a row, a card, an
image) that Kairo shouldn't impose a background/border on. `ContextMenuContent`
composes Base UI's `ContextMenu.Portal` > `ContextMenu.Positioner` +
`ContextMenu.Popup` in one call. Suppressing the browser's native menu,
opening on right click or a touch/long press, `role="menu"`/`"menuitem"`,
roving-focus keyboard navigation, and Escape/outside-press dismissal all come
from `@base-ui/react`.

<Callout title="Cursor positioning, not trigger positioning" type="info">
  Unlike `Select`/`Popover`, a context menu has no element for its popup to
  point at — it opens at wherever the pointer was when the user right-clicked
  (or where their finger was on a long press). Because of that,
  `ContextMenuContent`'s `side`/`align`/`sideOffset` are left **fully
  undefined by default**, unlike `SelectContent`'s: Base UI already supplies
  context-menu-aware defaults (`align: 'start'`, `sideOffset: -5`) that snug
  the popup's corner up against the cursor instead of floating a visible gap
  away from it. Only pass these props to override that behavior. There's also
  no `ContextMenuArrow` part — a context menu has no trigger element left to
  point back at once it opens.
</Callout>

<Callout title="A disabled trigger looks disabled too" type="info">
  Base UI's `ContextMenu.Trigger` genuinely honours `disabled` on
  `ContextMenu` — right-clicking or long-pressing it does nothing — but
  exposes no attribute for that state at all: its trigger state is just
  `{ open: boolean }`. Left alone, a disabled trigger used to still show the
  `context-menu` cursor, and right-clicking it popped the *browser's* native
  menu instead of doing nothing — worse than no fix at all. `ContextMenuTrigger`
  now stamps its own `data-disabled`/`aria-disabled`, entirely Kairo's own
  addition, not Base UI's. Its `disabled` prop, left unset, automatically
  mirrors the nearest `ContextMenu`'s own `disabled`, so setting `disabled` on
  `ContextMenu` alone is enough for both the interaction and the look — see
  **Props** below.
</Callout>

`ContextMenuItem` renders `children` directly (no built-in indicator, unlike
`SelectItem`'s check mark — a menu command has no selected state) inside Base
UI's `ContextMenu.Item`. Lay out an icon and/or a keyboard-shortcut hint as
flex children of it, the way the demo above does — `.kairo-context-menu-item`
is a flex row with `justify-content: space-between` for exactly that.

## Props

### `ContextMenu`

| Prop           | Type                                     | Default | Description                                                                                                                                          |
| -------------- | ------------------------------------------ | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `open`         | `boolean`                                  | —       | Controlled open state. Use with `onOpenChange`.                                                                                                          |
| `defaultOpen`  | `boolean`                                  | `false` | Initial open state for an uncontrolled context menu.                                                                                                     |
| `onOpenChange` | `(open: boolean, eventDetails) => void`    | —       | Called when the menu opens or closes.                                                                                                                    |
| `disabled`     | `boolean`                                  | `false` | Prevents the trigger from opening the menu — right-clicking the trigger falls through to the browser's native context menu instead of being suppressed. Also fans out to `ContextMenuTrigger`'s own `disabled` (see below), so the trigger looks disabled (`data-disabled`/`aria-disabled`, dimmed with a `not-allowed` cursor) without setting it there too. |
| ...            | Additional props | — | All other props are forwarded through. Unlike a dropdown `Menu`, `modal`, `openOnHover`, `delay`, and `closeDelay` aren't exposed — omitted since they don't apply to a menu that opens from a cursor position rather than hover/press on a trigger. |

### `ContextMenuTrigger`

Renders Base UI's `ContextMenu.Trigger`, a `<div>`. Unlike `SelectTrigger`/
`PopoverTrigger`, this is **not a control that opens on click** — it only
reacts to a native `contextmenu` event (right click) or a touch long-press.

| Prop        | Type      | Default | Description                                                        |
| ----------- | --------- | ------- | -------------------------------------------------------------------------- |
| `disabled`  | `boolean` | —       | Dims the trigger and swaps its cursor to `not-allowed` via `data-disabled`/`aria-disabled` — entirely Kairo's own addition; Base UI exposes no such attribute. Left unset, mirrors the nearest `ContextMenu`'s own `disabled` automatically, so the common case only needs `disabled` set on `ContextMenu`. Pass it here explicitly only to decouple the visual state from the root's actual `disabled` — doing so without also disabling the root reproduces the mismatch this prop exists to fix. |
| `className` | `string`  | —       | Merged with the base `kairo-context-menu-trigger` class.                   |
| ...         | Additional props | — | All other props are forwarded to the underlying trigger element. |

### `ContextMenuContent`

Composes Base UI's `ContextMenu.Portal` > `ContextMenu.Positioner` +
`ContextMenu.Popup` in one call. The forwarded ref attaches to the popup
element. No arrow is rendered — see the callout above.

| Prop         | Type                                                                       | Default | Description                                                                          |
| ------------ | ----------------------------------------------------------------------------- | ------- | -------------------------------------------------------------------------------------------- |
| `side`       | `'top' \| 'bottom' \| 'left' \| 'right' \| 'inline-end' \| 'inline-start'`     | —       | Which side of the cursor to render the popup against. Left undefined, Base UI picks whichever side has room in the viewport. |
| `align`      | `'start' \| 'center' \| 'end'`                                                 | —       | How to align the popup relative to `side`. Base UI defaults this to `'start'` for a context menu. Only pass this to override it. |
| `sideOffset` | `number`                                                                       | —       | Distance in pixels between the cursor and the popup along `side`. Base UI defaults this to a small negative offset so the popup overlaps the cursor slightly. Only pass this to override it. |
| `finalFocus` | `boolean \| RefObject \| function`                                             | —       | Element to focus when the menu closes. Defaults to the trigger, or whichever element was focused before it opened. Accepts `false` (leave focus untouched), `true` (use the default), a `RefObject` (focus that element), or a function — called with the interaction type (`'mouse'`, `'touch'`, `'pen'`, `'keyboard'`) — that returns an element, `true`/`null` for the default, or `false`/`undefined` to do nothing. |
| `className`  | `string`                                                                       | —       | Merged with the base `kairo-context-menu-popup` class on the popup element.                  |
| ...          | Additional props | — | All other props are forwarded to the underlying popup element. |

### `ContextMenuItem`

| Prop           | Type                                     | Default | Description                                                        |
| -------------- | ------------------------------------------ | ------- | -------------------------------------------------------------------------- |
| `disabled`     | `boolean`                                   | `false` | Prevents the item from being selected.                                     |
| `closeOnClick` | `boolean`                                   | `true`  | Whether activating the item closes the menu.                               |
| `onClick`      | `(event) => void`                           | —       | Called when the item is activated (click, or Enter/Space while highlighted). |
| `className`    | `string`                                    | —       | Merged with the base `kairo-context-menu-item` class.                      |
| ...            | Additional props | — | All other props are forwarded to the underlying item element. |

### `ContextMenuGroup` / `ContextMenuGroupLabel`

`ContextMenuGroup` groups related items; `ContextMenuGroupLabel` is an
accessible label automatically associated with its parent `ContextMenuGroup`.

### `ContextMenuSeparator`

A visual divider between items/groups. Unlike `SelectSeparator`, this is
**not** hidden from the accessibility tree by default: the `menu` role
(unlike `listbox`) explicitly permits a `separator` child, so exposing Base
UI's default `role="separator"` here is valid ARIA. Pass `aria-hidden` to opt
back out.

All other props are forwarded to the underlying element on each corresponding
Kairo component.
