## Usage

```tsx
import {
  AlertDialog,
  AlertDialogTrigger,
  AlertDialogContent,
  AlertDialogTitle,
  AlertDialogDescription,
  AlertDialogClose,
} from '@kairo-ui/react';

export function Example() {
  return (
    <AlertDialog>
      <AlertDialogTrigger>Delete account</AlertDialogTrigger>
      <AlertDialogContent>
        <AlertDialogTitle>Delete account?</AlertDialogTitle>
        <AlertDialogDescription>This action cannot be undone.</AlertDialogDescription>
        <AlertDialogClose>Cancel</AlertDialogClose>
        <AlertDialogClose>Delete</AlertDialogClose>
      </AlertDialogContent>
    </AlertDialog>
  );
}
```

`AlertDialog` groups the parts (it renders no DOM element of its own — the
same as Base UI's `AlertDialog.Root`). `AlertDialogContent` composes Base
UI's `AlertDialog.Portal` > `AlertDialog.Backdrop` + `AlertDialog.Popup` in
one call, so you don't need to reach for those parts yourself. Focus
trapping, scroll locking and `role="alertdialog"` all come from
`@base-ui/react` — as does the defining behaviour: clicking the backdrop (or
anywhere else outside the popup) **never** dismisses an alert dialog, unlike
`Dialog`. Escape still closes it, since Base UI treats it as a deliberate
keyboard action equivalent to Cancel, not an accidental dismissal.

<Callout title="AlertDialog vs Dialog" type="info">
  Both sit on the same Base UI anatomy — in fact `AlertDialog.Backdrop`,
  `Portal`, `Popup`, `Title`, `Description` and `Close` are `Dialog`'s exact
  same components, just running in a different mode — but they answer
  different questions. **Dialog** is an ordinary modal: it's fine for the
  user to bounce out of it by clicking elsewhere, since nothing irreversible
  is at stake. **AlertDialog** interrupts the user to demand one of a small
  set of explicit decisions (confirm, cancel, acknowledge) before they can
  continue — reach for it for destructive or otherwise consequential actions
  ("Delete this project?", "Discard unsaved changes?"), where silently
  dismissing the choice by clicking outside it would be a mistake, not a
  valid answer. If you're just showing content or a form with no
  point-of-no-return decision attached, use `Dialog` instead.
</Callout>

## Props

### `AlertDialog`

| Prop           | Type                                                        | Default | Description                                                                                                                                            |
| -------------- | -------------------------------------------------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `open`         | `boolean`                                                       | —       | Controlled open state. Use with `onOpenChange`.                                                                                                              |
| `defaultOpen`  | `boolean`                                                       | `false` | Initial open state for an uncontrolled alert dialog.                                                                                                         |
| `onOpenChange` | `(open: boolean, eventDetails) => void`                         | —       | Called when the alert dialog opens or closes. `eventDetails.reason` is `'escapeKey'`, `'closePress'`, `'triggerPress'`, `'focusOut'`, or `'imperativeAction'` — never `'outsidePress'`, since that dismissal path is disabled entirely. |
| ...            | `AlertDialog.Root` props from `@base-ui/react/alert-dialog`     | —       | All other Base UI `AlertDialog.Root` props are passed through. Unlike `Dialog`, `modal` and `disablePointerDismissal` aren't exposed — they're fixed to `true` internally.  |

### `AlertDialogTrigger` / `AlertDialogClose`

Render a native `<button>` by default, or compose with another element via
`render` (e.g. `<AlertDialogClose render={<Button>Delete</Button>} />`).
There's no separate "action" part — use two `AlertDialogClose`s (one for
Cancel, one for the confirming action) and attach your own `onClick` to the
confirming one for any side effect that should run before it closes.

### `AlertDialogContent`

| Prop           | Type                                                    | Default | Description                                                                             |
| -------------- | ---------------------------------------------------------- | ------- | ---------------------------------------------------------------------------------------------- |
| `className`    | `string`                                                    | —       | Merged with the base `kairo-alert-dialog-popup` class on the popup element.                    |
| `initialFocus` | `boolean \| RefObject \| function`                          | —       | Element to focus when the alert dialog opens. See Base UI's `AlertDialog.Popup` docs.           |
| `finalFocus`   | `boolean \| RefObject \| function`                          | —       | Element to focus when the alert dialog closes. See Base UI's `AlertDialog.Popup` docs.          |
| ...            | `AlertDialog.Popup` props from `@base-ui/react/alert-dialog`| —       | All other Base UI `AlertDialog.Popup` props are passed through.                                |

### `AlertDialogTitle` / `AlertDialogDescription`

Render an `<h2>`/`<p>` respectively, linked to the popup via
`aria-labelledby`/`aria-describedby` automatically.
